# Praxis — agency security model

Praxis is the only KAINE module that can change the host's state
(write files, send notifications, run shell commands). The other
modules read/perceive/think; Praxis acts. That makes the security
posture of Praxis load-bearing: if Praxis is compromised, KAINE can
modify your machine.

## Threat model

Praxis assumes:
- The operator controls what code runs as part of KAINE (no
  untrusted plugins yet).
- **Only the cycle's executive action-selection step (Volition) may
  direct Praxis.** Praxis realizes an `act` intent off `volition.out`
  only when the intent carries a valid provenance signature (see
  "Act-intent provenance" below); an intent forged by any other bus
  writer — Lingua is the most exposed, being LLM-output-driven — is
  dropped before any effector runs. Whatever the source, Praxis still
  refuses anything not explicitly allowed by the operator-configured
  whitelist.
- Logs may be inspected by humans or backed up; Praxis must NOT log
  payload contents (file body, shell stdout, notification body).

## Two enforced boundaries (defence in depth)

1. **Effector whitelist + sandbox — the PRIMARY gate.** The operator
   effector-enablement whitelist (empty by default) and each effector's
   own within-bounds enforcement (shell `CommandWhitelist`, file
   sandbox) decide what can run at all. This is the real, load-bearing
   control and is unchanged.
2. **Act-intent provenance — the SECOND boundary.** Inhibition (an
   inhibited snapshot yields no intent, so no effector acts) is a
   cognitive property of the legitimate path; it is made an *enforced*
   boundary at the Praxis interface by verifying that each `act` intent
   actually came from Volition. This closes the bus-injection path where
   a peripheral module synthesizes an `act` intent that never passed
   through Syneidesis's threshold or Volition's inhibition check.

## Act-intent provenance

KAINE runs single-process today (modules are asyncio tasks sharing one
Redis connection and one bus credential), so the bus cannot tell which
module published an event. Provenance is therefore cryptographic
(Mechanism B):

- A per-boot HMAC secret is generated by, and held only in, the cycle
  process. It is never published to the bus, written to disk, or logged.
- Volition attaches `sig = HMAC-SHA256(secret, canonical(kind, effector,
  params, run_id, seq))` to each `act` intent it emits.
- `Praxis._handle_intent` verifies `sig` (constant-time) **before**
  reading the effector name, constructing a request, or calling `act()`.
  A missing, invalid, or replayed signature drops the intent — no
  effector runs — and is audit-logged as `provenance_rejected`.
- Replay guard (in-process / per-boot): `(run_id, seq)` is signed, and
  Praxis rejects any `seq` at or below the highest it has already
  realized for that `run_id` (an O(1) high-water mark). This holds for
  the life of a boot: a full process restart rotates the secret +
  `run_id`, so signatures captured from a prior boot fail verification
  outright; a light module restart preserves the in-memory high-water
  mark (Praxis is not a heavy/externally-restarted module — it does not
  override `holds_external_resources()`, so Spot re-initializes the same
  instance). It is NOT an unconditional non-replay guarantee: if Praxis
  ever becomes a heavy module rebuilt on restart, a persisted/rotating
  replay window would be needed to keep the guarantee across a restart.
- Fail-closed: if enforcement is on (the default) but no secret was
  injected, Praxis refuses every `act` intent rather than pass silently.
- Threat scope: the secret is in-process, so a full compromise of the
  cycle process defeats it — but such an attacker already controls
  Volition. The boundary holds against the realistic threat: a
  compromised *peripheral* module. Per-process Redis ACLs are the
  direction once services split (out of scope here).

## Effector boundaries

| Effector | What it can do | What it cannot do |
|---|---|---|
| `file_write` | Write a UTF-8 string up to `max_file_bytes` to a path inside the configured sandbox (default `state/praxis/files/`). | Write absolute paths, paths containing `..`, symlinks resolving outside the sandbox. Write binary blobs (deferred). Delete files. |
| `notify` | Send a desktop notification via `notify-send` when present, otherwise append to a fallback log file. | Read existing notifications. Interact with any other system service. |
| `shell` | Run a command from the operator-configured `CommandWhitelist` with arg-pattern matching, a per-entry timeout, and per-entry working directory. | Run any command not in the whitelist. Use shell metacharacters. Read/write arbitrary files (each command's own permissions still apply). |

## Whitelist invariants

- The default whitelist ships empty. Until an operator adds entries,
  every shell action fails.
- Each entry pins a command (no metacharacters, no spaces) and one
  regex per arg position. The arg count must match exactly.
- Patterns SHOULD constrain to literal alphanumeric / hyphen / dot /
  underscore sets. `.*` and `[^x]*` patterns are accepted but the
  operator carries the risk.
- Timeouts default to 5 seconds. Long-running commands get explicit
  longer timeouts on a per-entry basis.

## Audit log

- Path: `state/praxis/audit.log` by default.
- Format: one JSON object per line; never partial-line writes
  (O_APPEND single-line write semantics).
- Fields: `timestamp`, `effector`, `request` (with content fields
  stripped), `success`, `elapsed_ms`, `error`, `blocked`,
  `provenance_rejected`.
- `blocked` (whitelist refusal) and `provenance_rejected` (forged/
  unsigned/replayed intent) are DISTINCT categories: a policy refusal
  vs. a security event at the provenance boundary.
- Content fields explicitly stripped before logging: `content`,
  `body`, `stdout`. A provenance-rejected record logs only the effector
  name and a generic reason — never the signature or params.

## Bus event posture

- `praxis.action` events carry only `effector`, `success`,
  `elapsed_ms`, `error`, `blocked`, and (on a provenance rejection)
  `provenance_rejected`. Nexus's privacy boundary is honored: the
  diagnostics view never sees what was acted on, only that an
  action ran (or was rejected at the provenance boundary).

## What Praxis cannot do (and why)

- **Output audio.** Phase 5.3 Chatterbox handles that.
- **Run a docker container, modify systemd units, or write to
  `/etc`.** Out of scope for v1; would need a separate effector
  with its own threat-modeled whitelist.
- **Make HTTPS calls / interact with cloud services.** KAINE is
  all-local at runtime (build prompt §1 operating principles).
